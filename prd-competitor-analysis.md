# PRD — viral-factory M1.1: Competitor Analysis (TikTok)

Summary: Перший модуль viral-factory. Користувач задає нішу → система знаходить конкурентів у TikTok, збирає їхні відео з метриками, зберігає структуровано (дати + цифри), і окремим кроком аналізує, що і як заходить у конкурентів та які тренди.

Sources: [marketing-tech/viral-content-method.md](../../wiki/marketing-tech/viral-content-method.md) (метод), [architecture.md](architecture.md) (плагін), канал [startupfomo](https://t.me/s/startupfomo).

Last updated: 2026-06-23

Status: **draft PRD** (до реалізації).

---

## 0. Контекст і місце в системі

Модуль **M1.1** треку [Marketing Tech](../../wiki/marketing-tech/index.md) → плагін viral-factory. Це **перший крок** повного пайплайну: до пошуку ідеї (`discover-niche`) і відтворення відео (`recreate-video`) ми спершу будуємо **карту конкурентного контенту** в ніші.

Складається з **двох явно розділених кроків** (як просив користувач):
- **Крок A — Збір** (детермінований): ніша → конкуренти → їхні відео → збагачення → структуроване збереження з датами й цифрами.
- **Крок B — Аналіз** (LLM поверх збережених даних): що/як заходить + тренди.

> Відмінність від Factory/Ares: Ares мапить **ринкових** конкурентів навколо *болю* (Reddit/Quora). Тут — **контентні** конкуренти в *TikTok-стрічці* навколо *формату й дистрибуції*. Інший вхід, інші дані.

---

## 1. Problem Statement

Щоб запускати B2C-апи через віральну органіку, треба знати, **який контент уже працює в ніші** — хто конкуренти в стрічці, що вони знімають, які формати/хуки залітають і куди рухається тренд. Зараз цього знання немає: ресерч робиться вручну, безсистемно, без цифр і без історії, тож рішення «який формат відтворювати» приймаються наосліп. Ціна невирішення — злив часу й бюджету на контент-гіпотези, які ринок уже відкинув.

---

## 2. Goals

1. **Задати нішу за <2 хв** і отримати підтверджений список TikTok-конкурентів без ручного пошуку.
2. **Зібрати ≥300 відео** по ніші з повними метриками за один прогін.
3. **100% зібраних відео** мають структуровані метадані з датою публікації і цифрами (views/likes/shares/saves/comments) — придатні для машинного аналізу.
4. **Дані зберігаються зі snapshot-датою**, щоб повторні прогони давали **динаміку** (velocity, що зростає / що насичується).
5. **Крок аналізу видає actionable-картину** «що і як заходить + тренди» з прив'язкою до конкретних відео-доказів — за <5 хв на готових даних.

---

## 3. Non-Goals

1. **Генерація відео** — це `recreate-video` (M1.4), поза цим PRD.
2. **Product spec / вибір ідеї** — це `spec-from-niche`, окремий крок.
3. **Instagram / YouTube** — лише TikTok у v1 (мультиплатформність — P2).
4. **Real-time моніторинг** — збір on-demand або за розкладом, не стрім.
5. **Ринковий конкурентний аналіз болю** — це Factory/Ares; тут лише контент у TikTok.
6. **Атрибуція виручки на ролик** — TikTok Shop revenue входить у `discover-niche` як сигнал ніші; тут опційне збагачення (P1), не ядро.

---

## 4. Users

Внутрішній інструмент. Основний користувач — **Growth PM / оператор Factory** (Артем): задає нішу, переглядає й підтверджує конкурентів, читає аналітичний звіт. Вторинний — **контент-команда**, що бере winning-формати в роботу.

---

## 5. User Stories

**Крок A — Збір:**
- Як оператор, я хочу **задати нішу** (keywords + опційно seed-акаунти), щоб не шукати конкурентів вручну.
- Як оператор, я хочу, щоб система **знайшла й запропонувала список конкурентів** у TikTok, ранжований за релевантністю і охопленням, щоб я підтвердив фінальний перелік (human gate).
- Як оператор, я хочу, щоб для кожного конкурента зібрався **перелік його відео** з повними метриками й датами.
- Як оператор, я хочу, щоб дані **зберігались структуровано в проєкті** (JSON + читабельний індекс), щоб до них можна було повертатись і порівнювати в часі.

**Крок B — Аналіз:**
- Як оператор, я хочу окремою командою **запустити аналіз** уже зібраних даних, щоб не змішувати дорогий збір із дешевим переаналізом.
- Як оператор, я хочу бачити **які формати/хуки заходять** (ранг за outlier-перформансом) з прикладами-відео.
- Як оператор, я хочу бачити **тренди**: що зростає, які звуки/хештеги набирають, з якою періодичністю постять конкуренти.
- Як оператор, я хочу бачити **whitespace** — формати/кути, які конкуренти НЕ використовують.

## 6. Архітектура потоку

### Крок A — Збір (детермінований)

```
ніша (config) 
  → A1. Competitor Discovery   — keyword/hashtag search → групування за автором → ранг
  → [HUMAN GATE: підтвердити список конкурентів]
  → A2. Video Harvest          — по кожному конкуренту тягнемо відео + метрики
  → A3. Enrichment             — outlier-score, save/share-rate, novelty-прапори (опц. транскрипт)
  → A4. Store Snapshot         — JSON зі snapshot_date + читабельний індекс
```

**A1. Competitor Discovery.** Скрапимо TikTok за keywords/hashtags ніші (+ seed-акаунти з конфігу). Групуємо знайдені відео за автором. Ранжуємо авторів за: к-сть niche-відео × сумарне/медіанне охоплення × commercial-сигнал (є посилання на апп/Shop). Output — кандидати в конкуренти. **Human gate**: оператор лишає/прибирає.

**A2. Video Harvest.** Для підтверджених конкурентів тягнемо їхні відео (ліміт N на акаунт, напр. останні 50–100) з повними метриками.

**A3. Enrichment.** Рахуємо похідні: `outlier_score = views / author_median`, `save_rate`, `share_rate`, `engagement_rate`, прапор `sound_is_trending`. Опційно (P1) — транскрипт + візуальна декомпозиція (делегується агенту `video-analyst`, див. архітектуру).

**A4. Store Snapshot.** Записуємо з `snapshot_date`. Повторний прогін = новий snapshot → можливість рахувати динаміку.

### Крок B — Аналіз (LLM поверх збережених даних)

Окрема команда. Читає останній (або два для динаміки) snapshot → видає звіт (розділ 8).

## 7. Модель даних і зберігання

**Розкладка в репо продукту:**
```
artifacts/viral-factory/niches/{slug}/
  config.snapshot.json           # копія ніша-конфігу на момент прогону
  competitors.json               # список конкурентів
  videos/{snapshot_date}.json    # відео + метрики на дату збору  ← snapshot
  analysis/{date}.md             # звіти кроку B
  index.md                       # людиночитабельний огляд (хто, скільки, коли)
```

**Competitor (запис):**
```json
{
  "handle","display_name","url","followers","account_created",
  "niche_relevance_score","video_count_scraped","median_views",
  "has_app_link","has_shop","confirmed_by_human": true, "first_seen": "YYYY-MM-DD"
}
```

**Video (запис, у snapshot):**
```json
{
  "video_id","competitor_handle","url","posted_at","scraped_at","duration_sec",
  "views","likes","comments_count","shares","saves",
  "caption","hashtags":[],"sound_id","sound_name","sound_is_trending",
  "outlier_score","save_rate","share_rate","engagement_rate",
  "decomposition": null   // заповнюється на P1 агентом video-analyst
}
```
> **Дати й цифри — first-class.** Кожен відео-запис має `posted_at` (коли вийшло) і `scraped_at` (коли зняли метрики) — це основа для трендів.

---

## 8. Крок B — що саме виводимо («що і як заходить + тренди»)

Звіт `analysis/{date}.md`:

**Що заходить (формати):**
- Топ-формати за медіанним `outlier_score` (faceless / talking-head / screencast / POV / duel…) з прикладами-відео.
- Топ хук-патерни (типові перші 3 сек, текст-оверлеї) — потребує P1-декомпозиції.
- Novelty-елементи, що повторюються у winner'ів.

**Як заходить (механіка):**
- Розподіл `save_rate` / `share_rate` по формату — що корисне vs що вірусне.
- Структури сценаріїв winner'ів (hook→problem→demo→CTA).

**Тренди:**
- **Within-snapshot:** розподіл `posted_at` топ-відео → що залітало останні 7/30 днів.
- **Cross-snapshot (≥2 прогони):** velocity — які відео/звуки/хештеги зростають між знімками; що насичується.
- Тренд-звуки і тренд-хештеги (частота + сумарне охоплення).
- Cadence: скільки постить кожен конкурент на тиждень.

**Whitespace:**
- Формати/кути, відсутні в конкурентів — гіпотези для нашого контенту.

**Per-competitor профілі:** розмір акаунту, медіана, найкращий формат, періодичність.

## 9. Requirements

### Must-Have (P0)
1. **Ввід ніші через конфіг** (keywords, опц. seed-акаунти, пороги, ліміти).
   - AC: Given валідний `config.yaml`, When запуск A, Then використовуються його keywords/пороги; за відсутності обов'язкових полів — зрозуміла помилка.
2. **Competitor Discovery + human gate.**
   - AC: система повертає ранжований список кандидатів; оператор може видалити/додати; підтверджений список зберігається в `competitors.json` з `confirmed_by_human`.
3. **Video Harvest з повними метриками й датами.**
   - AC: для кожного конкурента зібрано до N відео; кожне має `posted_at`, `scraped_at`, views/likes/shares/saves/comments, caption, hashtags, sound.
   - AC (negative): відео без обов'язкових полів логуються як skipped, не ламають прогін.
4. **Enrichment похідними метриками.**
   - AC: кожне відео має `outlier_score`, `save_rate`, `share_rate`, `engagement_rate`.
5. **Структуроване збереження зі snapshot-датою.**
   - AC: дані лягають у розкладку розділу 7; повторний прогін створює **новий** `videos/{snapshot_date}.json`, не перезаписує старий.
6. **Окрема команда кроку B**, що читає збережені дані й генерує `analysis/{date}.md` з розділами 8 (без трендів cross-snapshot, якщо snapshot один).
   - AC: звіт містить топ-формати, метрики по формату, within-snapshot тренди, whitespace, per-competitor профілі; кожне твердження посилається на конкретні video_id/URL.

### Nice-to-Have (P1)
7. **Візуальна декомпозиція відео** (`video-analyst`): hook-патерни, structure, novelty — наповнює `decomposition`.
8. **Cross-snapshot тренди** (velocity) при ≥2 прогонах.
9. **TikTok Shop збагачення** конкурентів-продавців (revenue-сигнал).
10. **Дедуплікація** конкурентів між повторними прогонами (`first_seen`).

### Future Considerations (P2)
11. Мультиплатформність (Instagram Reels, YouTube Shorts) — схеми лишити сумісними.
12. Розклад автоматичних прогонів (snapshot щотижня) для безперервного тренд-трекінгу.
13. Алерти на нові вірусні відео конкурента.

---

## 10. Success Metrics

**Leading:**
- Time-to-competitor-list: <2 хв від запуску A до human gate.
- Coverage: ≥300 відео / ≥10 конкурентів за прогін.
- Data completeness: ≥95% відео з повним набором обов'язкових полів.
- Time-to-insight: крок B <5 хв на готових даних.

**Lagging:**
- % winning-форматів зі звіту, узятих у `recreate-video`, що дали outlier у нас (зв'язок аналіз→результат).
- Скорочення ручного ресерч-часу на нішу (baseline: години → цільове: <30 хв оператора).

## 11. Open Questions

- **[eng]** Який саме Apify-актор для discovery за keyword+hashtag дає автора з median_views? Підтвердити через `search-actors`. (blocking для A1)
- **[eng]** TikTok не завжди віддає `saves` публічно — чи доступне поле через обраний актор? Якщо ні — save_rate стає P1. (blocking для P0-#4 у частині saves)
- **[data]** Поріг «niche_relevance» для авто-ранжування конкурентів — калібрувати на першій ніші.
- **[eng]** Ліміт N відео на акаунт: фіксований чи адаптивний від розміру акаунту?
- **[legal]** ToS/зберігання scraped-даних — межі для внутрішнього використання (перегукується з розділом 8.1 архітектури).

## 12. Timeline / Phasing

- **Фаза 1 (P0):** Кроки A1–A4 + базовий крок B без декомпозиції та cross-snapshot. Валідація на одній реальній ніші (напр. parental control / safety).
- **Фаза 2 (P1):** video-analyst декомпозиція + cross-snapshot velocity + Shop-збагачення.
- **Фаза 3 (P2):** мультиплатформа + розклад + алерти.

Залежності: спільна з усім viral-factory залежність на підтверджені Apify-актори (open question) і на схеми з [architecture.md](architecture.md).
